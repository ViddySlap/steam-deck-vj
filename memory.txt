Project
- Repo: steam-deck-vj
- Purpose: Steam Deck inputs are converted into stable Action events and sent over UDP to a Windows receiver, which emits MIDI for Resolume.
- Reliability requirement: show-safe behavior is more important than cleverness.
- GitHub repo name is `steam-deck-midi` even though the local folder name remains `steam-deck-vj`.

Current validated state
- Steam Deck sender preset workflow exists and launches correctly.
- Learn Wizard still uses the older xinput-based capture path and was intentionally left unchanged during sender runtime fixes.
- Deck sender runtime input backend was replaced with direct X11/XI2 raw key listening in Python via ctypes against libX11/libXi.
- Sender now listens to the configured XI2 device id directly instead of parsing `xinput test` text output.
- Sender emits one `down` on raw key press and one `up` on raw key release.
- Sender sends heartbeat packets while any key is held.
- Windows receiver now accepts heartbeat packets and uses them only to refresh sender liveness.
- Receiver still times out and releases held MIDI state if sender heartbeats stop unexpectedly.
- Windows receiver now supports separate MIDI output/input ports for Resolume:
  - receiver writes control output to `DECK_IN`
  - receiver listens for selected Resolume feedback on `DECK_OUT`
- Resolume feedback handling is validated for tracked `macro_cc` parameters:
  - matching fade feedback is ignored during active fades
  - manual Resolume movement during a fade cancels the fade and wins
- False manual-override detection from near-match feedback was fixed by adding a small feedback tolerance.
- R-pad horizontal clip scrolling now uses stateless relative CC output on CC 47:
  - `R_PAD_RIGHT` repeats value `1` while held
  - `R_PAD_LEFT` repeats value `127` while held
  - `R_PAD_RIGHT_LONG_PRESS` and `R_PAD_LEFT_LONG_PRESS` were restored to their original note mappings
- `DPAD_DOWN` click and long press are now intentionally split:
  - `DPAD_DOWN_LONG_PRESS` remains the Layer 1 opacity `macro_cc` fade/toggle on CC 20
  - `DPAD_DOWN` is now an independent note trigger on note 98 for manual Resolume mapping
- Left-pad long press now uses a staged dual-channel transition macro:
  - immediate Channel 1 modifier note on
  - delayed Channel 2 trigger note on after `macro_delay_ms`
  - Channel 1 modifier note off after `modifier_hold_ms`
- Left-pad staged transition macro hold now refreshes from either short navigation click while active:
  - `L_PAD_LEFT` refreshes the active modifier hold
  - `L_PAD_RIGHT` refreshes the active modifier hold
  - no extra modifier note-on is sent during refresh; only the release deadline moves

Architecture decisions
- Do not reintroduce repeated `down` messages for held buttons.
- Heartbeat is the chosen design for long holds because it preserves disconnect safety.
- Timeout must remain in the receiver as a fail-safe against stuck notes if sender/network dies mid-hold.
- Direct XI2 input is preferred over shelling out to `xinput` because the subprocess/text path proved nondeterministic and unsafe.
- Learn Wizard was not migrated to direct XI2 yet; keep that scope separate unless explicitly requested.
- Separate Resolume input/output loopMIDI ports are the chosen design because they avoid MIDI feedback loops while still allowing selective state feedback.
- Cache/feedback remains intentionally limited to the tracked `macro_cc` subset; relative encoder-style CCs stay stateless.
- Keep receiver behavior small and composable:
  - `macro_cc` for cached fade/toggle controls
  - `relative_cc` for stateless encoder-style scrolling
  - simple `note` for manual Resolume triggers
  - staged note macro for ordered multi-message transition workflows
- For left-pad transition workflows, treat the Channel 1 modifier as a leased hold owned by the receiver; long press starts it and either short left/right click can extend it.

Intentional MIDI mapping conflicts
- Do not remove or "fix" the following shared MIDI identities unless Ben explicitly asks to resolve that particular intentional conflict. These are design choices, not accidents.
- `DPAD_UP` and `DPAD_UP_LONG_PRESS` intentionally share CC 22 on channel 1 (`channel: 0` in config).
  Reason: click and long press are two gestures for the same Layer 2 opacity macro target, so they must point at the same CC identity.
  Resolve only if Ben explicitly decides those gestures should target different Resolume mappings.
- `DPAD_RIGHT` and `DPAD_RIGHT_LONG_PRESS` intentionally share CC 21 on channel 1 (`channel: 0` in config).
  Reason: click and long press are two gestures for the same Layer 1/right-side opacity macro target, so they must stay paired.
  Resolve only if Ben explicitly wants click vs long press split for that control.
- `DPAD_LEFT` and `DPAD_LEFT_LONG_PRESS` intentionally share CC 23 on channel 1 (`channel: 0` in config).
  Reason: click and long press are two gestures for the same left-side opacity macro target.
  Resolve only if Ben explicitly wants that macro pair broken apart.
- `R_PAD_LEFT` and `R_PAD_RIGHT` intentionally share CC 47 on channel 1 (`channel: 0` in config).
  Reason: they emulate a single relative encoder, where left sends value 127 and right sends value 1 on the same CC identity.
  Resolve only if Ben explicitly abandons the shared relative-encoder design.
- `START` raw learn output on channel 3 and ABXY layer lamps on channels 1/2 intentionally share CC 78.
  Reason: Ben wants to MIDI-learn once from the physical START button, then only change the channel in Resolume without changing the CC number.
  Resolve only if Ben explicitly changes the learn workflow or wants dedicated lamp CCs.
- `SELECT` raw learn output on channel 3 and trigger-layer lamps on channels 1/2 intentionally share CC 79.
  Reason: same learn workflow as START; raw learn path and authoritative layer-state outputs must keep the same CC identity across different channels.
  Resolve only if Ben explicitly changes that workflow.
- The layer lamp outputs on channels 1 and 2 are authoritative state-machine outputs and should not be treated as duplicates of the raw channel 3 learn messages.
  Reason: same CC number across different channels is intentional here; the channel split is the whole design.
  Resolve only if Ben explicitly replaces the three-channel layer-indicator strategy.

  - important: WSL IP changes over time; SSH may fail until the firewall rule remote address is updated.
